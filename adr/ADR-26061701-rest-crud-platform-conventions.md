# ADR-26061701: REST CRUD — платформенные соглашения

## Статус

✅ **Принято**

## Решение (кратко)

Все CRUD-микросервисы платформы с REST API следуют единому профилю HTTP-кодов, соглашению об именовании полей JSON и БД, пагинации списков и оформлению ошибок. В markdown-спецификациях (`*-controller.md`, `REST.md`, `readme.md`) **не дублируют** эти правила — достаточно ссылок на данный ADR и связанные ADR жизненного цикла записей.

**Зачем:** убрать повторяющиеся преамбулы в спецификациях, сохранить одну точку правды для code review и реализации.

---

## Контекст

Спецификации микросервисов в `documentation/specifications/microservices/` содержат секцию «Общие соглашения REST» с идентичным текстом в каждом файле контроллера: таблица HTTP-кодов, правило camelCase/snake_case, описание пагинации. Это дублирование усложняет сопровождение: при изменении платформенного правила приходится править десятки файлов, а расхождения между спеками уже возникают (например, разные формулировки 409).

Контракты проектируются до кода (ADR-26053102); OpenAPI уже содержит `$ref` на `InternalServerError` и параметры `PaginationLimit`/`PaginationOffset`. Платформенные соглашения логично зафиксировать в ADR, как сделано для логического удаления (ADR-26060701) и фильтров по дате (ADR-26061501).

## Решение

### 1. Профиль `crud-default`

Применяется ко **всем** CRUD-сервисам с REST API, если в спецификации сервиса явно не указан другой профиль (см. п. 4).

#### 1.1. Коды HTTP и тела ответов

| Код | Когда | Тело ответа |
| --- | ----- | ----------- |
| 200 | Успех | JSON по схеме операции (для `DELETE` — без тела) |
| 400 | Некорректный path, query или тело запроса | — |
| 404 | Запись не найдена или логически удалена ([ADR-26060701](./ADR-26060701-soft-delete-deleted-at.md)) | — |
| 409 | Конфликт уникальности или недопустимое состояние | — |
| 500 | Непредвиденная ошибка сервера | JSON [InternalServerError] (`message` + `stackTrace`) |

- Коды 400, 404, 409 — **без тела** (если в OpenAPI сервиса не зафиксировано иное для конкретной операции).
- Схема `InternalServerError` — в `components.schemas` OpenAPI сервиса; операции ссылаются через `$ref: "#/components/responses/InternalServerError"`.
- Дополнительные причины 400/404/409 **конкретной операции** описываются в спецификации контроллера как **дельты** (см. п. 5), без копирования всей таблицы.

#### 1.2. Именование полей JSON и БД

- Поля JSON в запросах и ответах — **camelCase**.
- Колонки PostgreSQL — **snake_case**.
- Имена optional query-параметров списков совпадают с полями JSON ответа (`createdAt` → `created_at`, `isActive` → `is_active`).

#### 1.3. Пагинация GET-списков

| Параметр | Тип | Ограничения | По умолчанию |
| -------- | --- | ----------- | ------------ |
| `limit` | integer | 1–100 | 20 |
| `offset` | integer | ≥ 0 | 0 |

Ответ списка — объект с полями:

- `total` — число записей **до** применения `limit`/`offset`;
- `limit` — фактически применённый limit;
- `offset` — фактически применённый offset;
- `list` — массив элементов по схеме операции.

В OpenAPI: переиспользовать `components.parameters.PaginationLimit` и `PaginationOffset` (или эквивалентные `$ref` в рамках сервиса).

Платформа **не задаёт** порядок сортировки по умолчанию — см. п. 1.6.

#### 1.4. Логическое удаление

Семантика `deleted_at` — по [ADR-26060701](./ADR-26060701-soft-delete-deleted-at.md). В преамбуле спецификации — **только ссылка**, без копирования текста про видимость записи и фильтр `deleted_at IS NULL`.

#### 1.5. Отбор по дате и времени

Query-параметры отбора по `timestamp with time zone` — по [ADR-26061501](./ADR-26061501-rest-timestamp-range-query-filters.md). В преамбуле — **только ссылка**; в таблице параметров списка — краткая отсылка к ADR для соответствующего поля.

#### 1.6. Сортировка GET-списков

Применяется ко **всем** сервисам с пагинированными GET-списками по данному ADR (включая профиль `validation-json-400`, например `scheduled-events/scheduler`).

| Параметр | Тип | Поведение |
| -------- | --- | --------- |
| `orderBy` | string (optional) | Имя поля **как в DTO** (camelCase). Одно поле. Допустимые значения — поля, по которым на этом эндпоинте есть query-фильтр (кроме служебных: `asOfDate`, `activeOnly`, `archivedOnly`, path-параметры). |
| `asc` | флаг (optional) | Присутствует в query string **без значения** (`?asc`). Направление ASC. |
| `desc` | флаг (optional) | Аналогично (`?desc`). Направление DESC. |

Правила:

- Без `orderBy` → `ORDER BY` в SQL **не добавляется**.
- `orderBy` без `asc`/`desc` → `ORDER BY` **не добавляется**.
- `orderBy` и ровно один из `asc`/`desc` → сортировка по указанному полю.
- Оба флага `asc` и `desc` → **HTTP 400**.
- Неизвестное или недопустимое для эндпоинта значение `orderBy` → **HTTP 400**.
- Флаги `asc`/`desc` без `orderBy` → игнорируются (без 400).

В OpenAPI: переиспользовать `components.parameters.OrderBy`, `SortAsc`, `SortDesc` (или эквивалентные `$ref` в рамках сервиса).

#### 1.7. Рекомендуемые зависимости для CRUD-сервисов

Для новых и дорабатываемых CRUD-сервисов использовать следующие базовые зависимости:

| Зависимость | Для чего |
| ----------- | -------- |
| `com.vimpishsnorfle.rest:<service-artifact>` | Generated OpenAPI-контракты: интерфейсы `*Api`, request/response DTO |
| `com.vimpishsnorfle.jooq:<schema-artifact>` | Generated jOOQ-модели БД: таблицы, `Record`, типобезопасные SQL-операции |
| `com.vimpishsnorfle.jooq:jooq-query-support` | Общие runtime-helper'ы для листингов: сортировка `orderBy/asc/desc`, timestamp-range, типовые `Condition` |
| `com.vimpishsnorfle.crud:crud-web-error-support` | Общий набор `BadRequest/NotFound/Conflict` и базовый `GlobalExceptionHandler`-шаблон |
| `org.springframework.boot:spring-boot-starter-web` | REST runtime (контроллеры, валидация входных данных, сериализация HTTP) |
| `org.springframework.boot:spring-boot-starter-jooq` | Интеграция Spring + jOOQ (`DSLContext`, транзакции, data-access слой) |
| `org.springframework.boot:spring-boot-starter-validation` | Bean Validation для query/path/body параметров |
| `com.vimpishsnorfle.database-test` (Gradle plugin) + `com.vimpishsnorfle.test:database-test-support` | Единый тестовый PostgreSQL-контур через prebaked image и Testcontainers |

Примечание:

- Для сервисов с Kafka дополнительно подключаются generated messaging-артефакты `com.vimpishsnorfle.messaging:*` и runtime-библиотеки по роли сервиса.
- Версии артефактов фиксируются в `build.gradle.kts` конкретного сервиса и должны быть совместимы между generated REST/jOOQ и схемой БД.

### 2. Профиль `lookup`

Сервисы **только чтения** справочных данных без операций delete (например, `classifiers-crud`):

| Код | Когда | Тело |
| --- | ----- | ---- |
| 200 | Успех | JSON по схеме |
| 400 | Некорректный path или query | — |
| 404 | Lookup — запись не найдена | — |
| 409 | Lookup — найдено более одной подходящей записи | — |
| 500 | Непредвиденная ошибка | JSON [InternalServerError] |

Логическое удаление (ADR-26060701) **не применяется**, если сервис не экспонирует delete и не использует `deleted_at`.

### 3. Профиль `validation-json-400`

Исключение для сервисов, где клиенту нужно тело при ошибке валидации:

| Код | Когда | Тело |
| --- | ----- | ---- |
| 400 | Некорректные параметры или тело | JSON [ErrorMessage] (`message`) |
| 404 | Запись не найдена | — |
| 409 | Конфликт уникальности или недопустимое состояние | — |
| 500 | Непредвиденная ошибка | JSON [InternalServerError] |

Профиль **явно указывается** в `index.md` / `readme.md` сервиса. Остальные правила `crud-default` (naming, пагинация при наличии списков) сохраняются.

### 4. Выбор профиля

| Профиль | Когда |
| ------- | ----- |
| `crud-default` | CRUD с delete, большинство сервисов |
| `lookup` | Только GET lookup без delete |
| `validation-json-400` | Явно зафиксировано в readme сервиса |

### 5. Оформление markdown-спецификаций

В `*-controller.md`, `REST.md`, `readme.md`:

1. **Не копировать** таблицу кодов HTTP, правило camelCase/snake_case и описание пагинации из данного ADR.
2. В начале файла (или в `index.md` сервиса для многофайловых сервисов) — перечень применимых ADR:

   ```markdown
   HTTP-контракт описан в [openapi.yaml](./openapi.yaml).

   Общие соглашения REST — [ADR-26061701](../../../ADR/ADR-26061701-rest-crud-platform-conventions.md) (профиль `crud-default`), [ADR-26060701](../../../ADR/ADR-26060701-soft-delete-deleted-at.md), [ADR-26061501](../../../ADR/ADR-26061501-rest-timestamp-range-query-filters.md).
   ```

3. Секция **«Дельты данного контроллера»** (или **«Дельты сервиса»**) — **только** отличия от платформы:
   - дополнительные причины 400/404/409 в таблице «Когда»;
   - доменные подразделы (архивация, привязка к версии — со ссылкой на профильный ADR);
   - сервисные query-параметры и бизнес-правила.

4. В алгоритмах операций допустимы inline-упоминания «HTTP 404 по ADR-26060701», «пагинация по ADR-26061701» без повторения числовых лимитов, если они не меняются.

### 6. OpenAPI

- Все операции CRUD-профиля перечисляют коды 200/400/404/409/500 согласно профилю (где применимо).
- `description` параметров пагинации и временны́х фильтров — отсылка к ADR-26061701 / ADR-26061501.
- Переиспользовать общие компоненты сервиса: `InternalServerError`, `PaginationLimit`, `PaginationOffset`, `OrderBy`, `SortAsc`, `SortDesc`.

## Аргументация

| Альтернатива | Почему не выбрана |
| ------------ | ----------------- |
| **Дублирование в каждой спеке** | Уже привело к расхождениям; дорого сопровождать |
| **Shared markdown без ADR** | Нет статуса архитектурного решения; слабее для review |
| **Отдельный ADR на HTTP, naming, pagination** | Избыточная дробность; правила используются только вместе |

## Последствия

### Позитивные

- Сокращение преамбул спецификаций на ~20–30 строк на файл.
- Единая точка правды для HTTP-кодов, naming и пагинации.
- Согласованность с ADR-26060701 и ADR-26061501 (ссылки вместо копий).

### Негативные

- Читатель спецификации должен открыть ADR для полной картины платформы (смягчается ссылками в `index.md` сервиса).
- Миграция существующих 15+ файлов — разовая работа.
