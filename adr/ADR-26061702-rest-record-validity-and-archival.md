# ADR-26061702: Действительность и архивация записей в REST

## Статус

✅ **Принято**

## Решение (кратко)

Запись в CRUD-сервисе может быть **логически удалена** (`deleted_at`, ADR-26060701) или **заархивирована / недействительна** по бизнес-правилам (`archived_at`, `archived_dt`, `enacted_dt`). Платформа использует **два профиля** REST-поведения для архивации; смешивать их в одной таблице или одном сервисе без явного указания профиля **запрещено**.

**Зачем:** убрать дублирование блоков «Заархивированная запись» и «Действительность на дату» в спецификациях; зафиксировать различие моделей Фабрики продуктов и «Валюты и ценности».

---

## Контекст

В спецификациях повторяются два семейства правил:

1. **Фабрика продуктов** — `archived_at timestamp with time zone`, nullable; заархивированная запись **видна** в GET; фильтр `archivedOnly`; PUT и повторная архивация блокируются.
2. **Валюты и ценности** — temporal validity через `enacted_dt` / `archived_dt` (или `archived_at` для курсов); query `asOfDate` и `activeOnly`; две подмодели хранения «активности».

ADR-26060701 уже разделяет `deleted_at` и `archived_*`, но не описывает REST-семантику архивации и temporal lookup. Спецификации копируют по 10–15 строк в каждый `REST.md` / `*-controller.md`.

## Решение

### 1. Общие правила

- `deleted_at` и поля архивации — **разные оси** (ADR-26060701, п. 6).
- Логически удалённая запись (`deleted_at IS NOT NULL`) **не существует** для API независимо от `archived_*`.
- Partial unique indexes учитывают оба предиката, где применимо.
- В markdown-спецификациях — **ссылка на профиль** данного ADR, без копирования полного текста правил.

### 2. Профиль A — `archived_at` nullable

**Применение:** Фабрика продуктов — `products`, `product_versions`, `party_types`, `participant_types` и аналогичные справочники с колонкой `archived_at` (NULL = не заархивировано).

#### 2.1. Семантика

| Состояние | `archived_at` | Видимость в GET-списках и lookup |
| --------- | ------------- | -------------------------------- |
| Активная | `NULL` | Да |
| Заархивированная | заполнено | **Да** (в отличие от deleted) |

#### 2.2. Query-параметр `archivedOnly`

Optional boolean в GET-списках:

- `true` — отбор `archived_at IS NOT NULL`;
- `false` — отбор `archived_at IS NULL`;
- не передан — фильтр по архивации **не накладывается** (если в спецификации операции не указано иное).

Отбор по моменту архивации — через `archivedAt` ([ADR-26061501](./ADR-26061501-rest-timestamp-range-query-filters.md)).

#### 2.3. REST-операции над заархивированной записью

| Операция | Разрешено |
| -------- | --------- |
| GET (список, по uuid) | Да |
| DELETE (логический, ADR-26060701) | Да |
| PUT | **HTTP 409** |
| Повторная архивация (POST …/archive) | **HTTP 409** |

Архивация: `UPDATE SET archived_at = now()` (и `updated_at = now()` при наличии колонки).

#### 2.4. Дельты сервиса (не платформа)

- **Продукт** с `archived_at IS NOT NULL`: для продукта доступны **только операции чтения**; любая мутация — HTTP 409 (описывается в `product-controller.md`).
- **Версия продукта**: жизненный цикл и иммутабельность — [ADR-26061401](./ADR-26061401-product-factory-version-immutability.md).

### 3. Профиль B — temporal validity (`enacted_dt` / `archived_dt`)

**Применение:** справочники подсистемы «Валюты и ценности» с периодом действия записи.

#### 3.1. Общая формула «действует на дату D»

`D` — тип `date`. Запись **действует на D**, если:

```text
enacted_dt <= D  AND  <условие активности archived_dt или archived_at>
```

Конкретное условие активности — по подварианту B1 или B2.

#### 3.2. Query-параметры

| Параметр | Назначение |
| -------- | ---------- |
| `asOfDate` | Проверка действительности на указанную дату; при передаче **`activeOnly` игнорируется** |
| `activeOnly` | Boolean, по умолчанию `true` — отбор только «активных» по правилам подварианта |

Если `asOfDate` **не** задан и `activeOnly=false`, отбор по срокам `enacted_dt` / `archived_dt` **не накладывается**.

#### 3.3. Подвариант B1 — `archived_dt` nullable

**Таблицы:** `currencies`, `precious_metals` (колонка `archived_dt`).

- **Активная** запись: `archived_dt IS NULL`.
- **Заархивированная:** `archived_dt` заполнен.
- `activeOnly=true` (default): `archived_dt IS NULL`.
- На дату `D`: `enacted_dt <= D AND (archived_dt IS NULL OR archived_dt > D)`.

#### 3.4. Подвариант B2 — `archived_dt` всегда заполнен

**Таблицы:** `currencies_classifier` и аналоги.

- Поле `archived_dt` **всегда** NOT NULL.
- **Действующая** запись: дата в далёком будущем (напр. `2099-12-31`).
- **Недействующая:** дата в прошлом (фактическое окончание).
- На дату `D`: `enacted_dt <= D AND archived_dt > D`.
- `activeOnly=true` (default): частный случай с `D = CURRENT_DATE`.

#### 3.5. Профиль B для курсов (`archived_at` на rates)

Сервисы курсов (`currencies-rates-crud`, `precious-metals-rates-crud`) используют **`archived_at`** (timestamptz), не `archived_dt`:

- **Активная** запись: `archived_at IS NULL`.
- `activeOnly=true` (default): `archived_at IS NULL`.
- Деактивированные записи могут возвращаться при `activeOnly=false` (GET по uuid и списки).

Сервисные правила immutability и replace (US-05) остаются в спецификации сервиса.

### 4. Оформление спецификаций

1. В `index.md` группы сервисов (напр. `currency-valuables/readme.md`) — одна ссылка на профиль B и подвариант группы.
2. В `REST.md` / `*-controller.md` — указание профиля A или B (B1/B2) **одной строкой**; сервисные дельты (напр. `currentRate`, US-05) — в секции «Дельты сервиса».
3. **Не копировать** формулы и таблицы профиля целиком, если они не меняются относительно ADR.

## Аргументация

| Альтернатива | Почему не выбрана |
| ------------ | ----------------- |
| **Один профиль archived_at для всех** | Не покрывает temporal validity справочников ОКВ |
| **Только в group readme без ADR** | Нет архитектурного статуса; дублирование между группами |
| **Полный перенос в ADR-26060701** | Разная семантика archived vs deleted; документ перегружен |

## Последствия

### Позитивные

- Единое описание двух моделей архивации.
- Сокращение преамбул currency-valuables и product-factory references.
- Явный запрет смешивать B1 и B2 в одной таблице без документирования.

### Негативные

- Новые сервисы должны явно выбрать профиль A, B1 или B2 при проектировании схемы.
