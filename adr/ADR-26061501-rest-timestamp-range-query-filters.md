# ADR-26061501: Отбор по диапазону даты/времени в REST-списках

## Статус

✅ **Принято**

## Решение (кратко)

В операциях **GET**-списков CRUD-сервисов отбор по полям типа `timestamp with time zone` задаётся **одним** optional query-параметром на колонку. Имя параметра — **camelCase**, как в JSON ответа (`createdAt`, `updatedAt`, `archivedAt`, `effectiveAt` и т.д.); в запросе к БД маппится на snake_case (`created_at`, `updated_at`, …).

Значение параметра — **строка** с разделителем `...` (три точки) и одним из трёх форматов:

| Формат строки | Условие на колонку `{column}` |
| ------------- | ------------------------------ |
| `${timestamptz}...` | `{column} >= ${timestamptz}` |
| `...${timestamptz}` | `{column} <= ${timestamptz}` |
| `${timestamptz1}...${timestamptz2}` | `{column} BETWEEN ${timestamptz1} AND ${timestamptz2}` |

`${timestamptz}` — корректная метка времени с часовым поясом (ISO 8601 / `timestamptz` PostgreSQL). Любой другой формат строки — **HTTP 400**.

Параметр **не передан** — отбор по соответствующей колонке **не накладывается**.

**Зачем:** единый компактный синтаксис фильтрации в URL без пар `*From`/`*To`; предсказуемое поведение во всех списках платформы.

---

## Контекст

CRUD-сервисы возвращают пагинируемые списки с фильтрами по query-параметрам (ADR-26053103). Поля аудита и жизненного цикла (`created_at`, `updated_at`, `archived_at`, `effective_at`) имеют тип `timestamp with time zone`.

Альтернативы — отдельные `createdAtFrom` / `createdAtTo` или повторное описание правил парсинга в каждой спецификации — приводят к расхождениям контрактов и дублированию в OpenAPI и markdown-алгоритмах.

Первое применение зафиксировано в [scheduled-events-scheduler](../specifications/microservices/common/scheduled-events/scheduler/readme.md); механика распространяется на все новые и существующие списки с отбором по дате/времени.

## Решение

### 1. Именование query-параметров

- Имя совпадает с полем JSON записи в camelCase: `createdAt`, `updatedAt`, `archivedAt`.
- Для иных временны́х полей — по тому же правилу (`effectiveAt` → `effective_at`).
- Пары `createdAtFrom` / `createdAtTo` **не используются** в новых контрактах.

### 2. Синтаксис значения

Разделитель диапазона — литеральная подстрока `...` (три символа точки).

Примеры (OpenAPI `components.examples`):

| Имя примера | Значение |
| ----------- | -------- |
| `PeriodAfter` | `2026-05-30T18:25:42.296Z...` |
| `PeriodBefore` | `...2026-05-30T18:25:42.296Z` |
| `Period` | `2026-05-30T18:25:42.296Z...2026-06-01T00:00:00.000Z` |

Границы диапазона **включительны** (`>=`, `<=`, `BETWEEN … AND …`).

### 3. Обработка в сервисе

1. Если query-параметр отсутствует — условие по колонке не добавляется.
2. Если присутствует — строка разбирается по таблице форматов выше; при успехе добавляется предикат к выборке.
3. При нераспознанном формате — **HTTP 400** (некорректные query-параметры).
4. Реализация — в общем слое разбора query (по возможности переиспользуемый между сервисами); в спецификации сервиса — **ссылка на данный ADR**, без дублирования правил парсинга.

### 4. OpenAPI

- Тип параметра: `string` (не `format: date-time` — значение не является одной меткой времени).
- В `description` — отсылка к ADR-26061501.
- Рекомендуется переиспользовать примеры `PeriodAfter`, `PeriodBefore`, `Period`.

### 5. Спецификации сервисов

В `*-controller.md` / `REST.md` / `readme.md`:

- подраздел «Отбор по дате и времени» в преамбуле **не создаётся** — достаточно одной ссылки на ADR в перечне общих соглашений ([ADR-26061701](./ADR-26061701-rest-crud-platform-conventions.md), п. 1.5) или в `index.md` сервиса;
- в таблице параметров списка — «отбор по `{field}` ([ADR-26061501](./ADR-26061501-rest-timestamp-range-query-filters.md))»;
- в алгоритме — «`{createdAt}` — [ADR-26061501] для поля `created_at`», без перечисления трёх форматов.

## Аргументация

| Альтернатива | Почему не выбрана |
| ------------ | ----------------- |
| Пара `*From` / `*To` | Два параметра на колонку; раздувает OpenAPI и URL |
| Один `format: date-time` | Не выражает открытые и двусторонние диапазоны |
| Дублирование в каждой спеке | Расхождения при изменении правил; нарушение DRY |

## Последствия

### Позитивные

- Единый синтаксис фильтрации по времени во всех REST-списках.
- Короткие URL и компактные OpenAPI-контракты.
- Одна точка правды для реализации парсера и code review.

### Негативные

- Нестандартный для «чистого REST» синтаксис строки; требует документации и примеров.
- Символ `...` в query должен корректно кодироваться клиентами (обычно без экранирования).

### Компромиссы

- Строковый DSL вместо нескольких типизированных параметров — осознанный обмен ради единообразия и краткости.
