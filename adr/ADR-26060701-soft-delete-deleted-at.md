# ADR-26060701: Логическое удаление через deleted_at во всех CRUD-сервисах

## Статус

✅ **Принято**

## Решение (кратко)

Во **всех CRUD-сервисах** платформы операция удаления записи — **логическая**, не физическая. SQL-оператор `DELETE FROM …` для пользовательских данных **запрещён**. Вместо этого выполняется `UPDATE … SET deleted_at = now()` (или эквивалент с текущим моментом времени в часовом поясе `timestamp with time zone`).


| Правило               | Суть                                                                                                                                                                                                                                     |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Удаление в API        | `deleted_at` заполняется текущим моментом времени; повторное удаление уже удалённой записи — HTTP 404                                                                                                                                    |
| Видимость записи      | Запись с `deleted_at IS NOT NULL` **не существует** для API, консьюмеров и внутренних read-операций CRUD-сервиса                                                                                                                         |
| Схема БД              | У каждой таблицы, над которой CRUD-сервис выполняет удаление, **обязательна** колонка `deleted_at timestamp with time zone`, по умолчанию `NULL`                                                                                         |
| Индексы и ограничения | Уникальные индексы, частичные ограничения и обычные индексы, обеспечивающие бизнес-уникальность или поиск по «активным» записям, **должны** включать условие `deleted_at IS NULL` (частичный индекс / partial unique index в PostgreSQL) |
| Чтение и изменение    | Все `SELECT`, `UPDATE` и операции «удаления» накладывают фильтр `deleted_at IS NULL`, если иное явно не зафиксировано в спецификации сервиса                                                                                             |


**Зачем:** единообразное поведение удаления, сохранение истории, возможность аудита и согласованность уникальных ограничений при повторном создании «удалённых» сущностей.

---

## Контекст

CRUD-сервисы платформы предоставляют REST (и при необходимости асинхронные) операции над справочниками и транзакционными сущностями в PostgreSQL (ADR-26060201, ADR-26060301). Контракты сервисов проектируются до кода (ADR-26053102).

Физическое удаление строк из БД создаёт проблемы:

- потеря истории и затруднение расследования инцидентов;
- нарушение ссылочной целостности и каскадных связей;
- невозможность повторно создать сущность с теми же бизнес-ключами при сохранённых уникальных индексах без `deleted_at IS NULL`;
- расхождение семантики HTTP `DELETE` (идемпотентное «исчезновение» ресурса) и фактического состояния данных.

Требуется зафиксировать единый подход для **всех** текущих и будущих CRUD-сервисов.

## Решение

### 1. Семантика удаления в CRUD-сервисе

- REST-операция `DELETE /…/{uuid}` (и эквиваленты в AsyncAPI) выполняет **логическое** удаление: `UPDATE … SET deleted_at = now()` (при наличии `updated_at` — также обновить его по соглашению модуля).
- Запись с заполненным `deleted_at` для внешнего потребителя **не существует**: списки её не возвращают, `GET` по `uuid` отвечает **HTTP 404**, повторный `DELETE` — **HTTP 404**.
- SQL `DELETE FROM <table>` в коде CRUD-сервиса для пользовательских данных **запрещён**. Исключения — только по одобренному RFC (например, техническая очистка служебных таблиц, не экспонируемых через CRUD API).

### 2. Колонка deleted_at в схеме БД

Для **каждой** таблицы, над которой CRUD-сервис выполняет операцию удаления:

```sql
deleted_at timestamp with time zone
```

- Тип: `timestamp with time zone` (согласован с остальными временны́ми полями платформы).
- Значение по умолчанию: `NULL` (запись активна).
- `NULL` — запись **не удалена**; непустое значение — момент логического удаления.
- Колонка добавляется **новой Flyway-миграцией** в репозитории `dbmigrations` подсистемы (ADR-26060301); изменение уже применённых миграций запрещено.
- Рекомендуется `COMMENT ON COLUMN`, фиксирующий семантику для ревьюеров и DBA.

Таблицы, к которым CRUD-сервис обращается **только на чтение** и не выполняет удаление, могут не иметь `deleted_at` — требование распространяется только на таблицы с операцией delete в контракте сервиса.

### 3. Индексы и ограничения уникальности

Любой уникальный индекс или ограничение, которое должно действовать **только среди активных (не удалённых) записей**, оформляется как **частичный** (partial) с предикатом, включающим `deleted_at IS NULL`.

Если у таблицы уже есть другие предикаты «активности» (например, `archived_dt IS NULL`), они **объединяются** в одном `WHERE` индекса:

```sql
create unique index currencies_char_code_uindex
    on currency_valuables.currencies (char_code)
    where archived_dt is null and deleted_at is null;
```

Обычные (неуникальные) индексы, предназначенные для поиска и сортировки **только по активным** записям, также создаются с `where deleted_at is null`, чтобы уменьшить размер индекса и соответствовать фильтрам запросов.

**Запрещено** оставлять глобальный `UNIQUE` без учёта `deleted_at`, если CRUD-сервис допускает логическое удаление и повторное создание сущности с теми же бизнес-атрибутами.

При добавлении `deleted_at` в существующую таблицу миграция **пересоздаёт** затронутые индексы: `DROP INDEX IF EXISTS …` → `CREATE … WHERE … deleted_at IS NULL`.

### 4. Доступ к данным в коде сервиса

Согласовано с ADR-26060201:

- В слое репозиториев (jOOQ) все выборки, обновления и операции удаления **по умолчанию** добавляют условие `.and(DELETED_AT.isNull)` (или эквивалент).
- Отдельный метод репозитория для доступа к «включая удалённые» записям допускается **только** при явной необходимости (админ-инструменты, фоновые задачи) и с одобрением в спецификации сервиса.

### 5. Спецификации и контракты

- В REST/AsyncAPI-спецификациях CRUD-сервиса с операцией delete — **ссылка** на данный ADR (и [ADR-26061701](./ADR-26061701-rest-crud-platform-conventions.md)); **не дублировать** текст про видимость записи и фильтр `deleted_at IS NULL` в каждом `*-controller.md` / `REST.md`.
- OpenAPI: операция delete описывает логическое удаление (`deleted_at = now()`), не физическое.
- Новые спецификации и миграции **не принимаются** без `deleted_at` и корректных partial-индексов на таблицах с delete.
- В `async.md` консьюмеров — краткая отсылка к ADR-26060701 для фильтра `deleted_at IS NULL`, без полного повторения раздела «Логическое удаление».

### 6. Связь с другими полями жизненного цикла

`deleted_at` и поля вроде `archived_at` / `archived_dt` — **разные** семантики:


| Поле         | Назначение                                                                                             |
| ------------ | ------------------------------------------------------------------------------------------------------ |
| `archived_*` | Бизнес-недействительность / архивация (запись может оставаться видимой в API в определённых сценариях) |
| `deleted_at` | Исключение записи из предметной области CRUD-сервиса; для API запись «не существует»                   |


Оба предиката могут присутствовать в одном partial unique index, если уникальность определяется только для записей, одновременно не заархивированных и не удалённых.

## Аргументация


| Альтернатива                     | Почему не выбрана                                                                                               |
| -------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| **Физический `DELETE`**          | Потеря истории; сложность с FK; конфликты уникальности при повторном создании; несовместимость с аудитом.       |
| **Флаг `is_deleted boolean`**    | Нет момента удаления для аудита и расследований; типично требует отдельного `deleted_at` для полноты картины.   |
| **Отдельная таблица «корзина»**  | Избыточная сложность для CRUD-справочников; дублирование схемы и JOIN-ов.                                       |
| **Локальное решение per-сервис** | Уже привело к расхождениям (часть таблиц без `deleted_at` в ранних миграциях); нужен единый стандарт платформы. |


**Логическое удаление через** `deleted_at` **выбрано**, потому что хорошо стыкуется с PostgreSQL partial indexes; сохраняет историю без усложнения модели; согласуется с идемпотентной семантикой REST DELETE (ресурс «исчезает» для клиента).

## Последствия

### Позитивные

- Единое поведение delete во всех CRUD-сервисах платформы.
- Сохранение данных для аудита, поддержки и восстановления при необходимости.
- Корректная уникальность: повторное создание сущности после логического удаления не блокируется старыми строками.
- Меньший размер рабочих индексов за счёт partial `WHERE deleted_at IS NULL`.
- Явная фиксация в ADR упрощает code review миграций и спецификаций.

### Негативные

- Рост объёма таблиц; периодическая архивация или purge удалённых записей — отдельное решение (не входит в данный ADR).
- Все запросы должны помнить о фильтре `deleted_at IS NULL` — риск утечки «удалённых» записей при ошибке в репозитории (смягчается code review и единым шаблоном репозитория).
- Миграции существующих таблиц требуют пересоздания индексов и синхронизации спецификаций.
- Усложнение partial unique indexes при нескольких предикатах активности (`archived_`*, `deleted_at`).

### Компромиссы

- Отказ от немедленного освобождения места в БД в пользу прозрачности и простоты контракта.
- Жёсткое требование к индексам увеличивает объём SQL в миграциях, но предотвращает трудноуловимые 409-конфликты в runtime.
